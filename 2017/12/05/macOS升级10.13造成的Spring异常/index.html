macOS升级10.13造成的Spring异常

发表于 2017-12-05 | 分类于 其他 | |

问题描述

SSM项目中，一段代码放到macOS 10.12系统下可以正常运行，然而放到macOS 10.13系统下则报错Nullpointer Exception。

异常信息

问题分析

从异常信息中可以看出，异常来自Spring创建TokenService的Bean时构造函数抛出了异常。TokenService类在创建时使用ApplicationContextAware的自定义实现类SpringContext来获取Spring上下文对象，并使用其getBean()方法来获取容器中的对象。

自定义SpringContext类如下：

在getBean方法中打断点发现，进入断点时静态成员变量context居然为null。

ApplicationContextAware接口的实现类会在该类被Spring实例化时由其为覆写方法setApplicationContext()的形参注入Spring上下文对象applicationContext。

这样看来，显然，调用SpringContext的getBean()时，Spring容器尚未实例化SpringContext类。

但该问题为何单单出现在macOS 10.13，而更早的系统以及Linux或Windows下就没有这个问题呢？

分析一下注解Bean的加载顺序：

注解Bean的创建机制是通过Classloader的getResource()来获取.class文件的路径，根据.class文件的本地排序顺序来加载.class文件并创建Bean。

问了下度娘，有小伙伴表示该加载顺序其实就是同一个目录下的首字母排序。要这么说起来，SpringContext原本确实应该在TokenService之前被Spring实例化。

macOS 10.13采用了新的文件系统APFS，猜测是该文件系统的底层排序机制的不同影响了Spring实例化Bean的顺序。

解决方案

网上提供的解决方案

该答主需要先加载的Bean按文件名排序在其依赖者后面，因此他修改了被依赖的Bean的名字，让它在按文件名排序时可以排到依赖者前面去。

显然，该解决方案不太适用于我们目前的情况。

采用的解决方案

为TokenService上加上@DependOn("springContext")注解，声明TokenService的Bean创建依赖于SpringContext的Bean创建，从而改变其加载顺序。问题解决。
