Gradle入门

发表于 2017-10-27 | 分类于 后端 | |

简介

  • 与Maven相同的是，Gradle本身只提供构建项目的框架，真正起作用的是plugin。构建Java项目或打war包等，都是通过Plugin实现的。

  • 与Maven不同，Gradle不提供内建的生命周期管理。只是java Plugin向Project中添加了许多Task，而Task依次执行。

  • 本身的领域对象主要有Project和Task。

  • 语法使用基于groovy的DSL。

    ​

领域对象

Project

  • Project为Task提供了执行上下文，所有的Plugin要么向Project中添加用于配置的Property，要么向Project中添加不同的Task

Task

  • 一个Task表示一个逻辑上较为独立的执行过程，比如编译Java源代码，拷贝文件，打包Jar文件，甚至可以是执行一个系统命令或者调用Ant。另外，一个Task可以读取和设置Project的Property以完成特定的操作。
  • task关键字其实本身也是一个属于Project的方法调用。
  • 对于每一个Task，Gradle都会在Project中创建一个同名的Property，所以我们可以将该Task当作Property来访问。

内置Task

  • Gradle在默认情况下为我们提供了几个常用的Task，比如查看Project的Properties、显示当前Project中定义的所有Task等。

    • 查看Project中可以执行的Task: gradle tasks 或 gradle tasks --all
    • 查看Project中具有的Properties: gradle properties

Properties

  • 自带巨多，以下列出常用的：

    • allprojects：所有的Project，即根项目和子项目
    • subprojects：子项目
    • buildDir: 构建结果的输出目录
    • rootDir：根项目路径
    • rootProject：根项目
    • projectDir： 项目目录（根或子各自的）

java Plugin

  • 和其他Plugin一样，java Plugin并没有什么特别的地方，只是向Project中引入了多个Task和Property。
  • 它在项目中引入了构建生命周期的概念，就像Maven一样。

BuildScript代码块：

  • buildscript中的声明是gradle脚本自身需要使用的资源。可以声明的资源包括依赖项、第三方插件、maven仓库地址等。而在build.gradle文件中直接声明的依赖项、仓库地址等信息是项目自身需要的资源。
  • radle在执行脚本时，会优先执行buildscript代码块中的内容，然后才会执行剩余的build脚本。
  • buildscript代码块中的repositories和dependencies的使用方式与直接在build.gradle文件中的使用方式几乎完全一样。唯一不同之处是在buildscript代码块中可以对dependencies使用classpath声明。该classpath声明说明了在执行其余的build脚本时，class loader可以使用这些你提供的依赖项。
